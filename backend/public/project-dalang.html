Konsep

Apa itu Project Dalang

Mesin narasi yang menjalankan map-reduce di atas LLM. Istilahnya pakai metafora wayang:

🎭 Dalang

Si orchestrator. Menerima request, menentukan strategi, menyebar worker, lalu menyatukan hasil. Tidak menulis sendiri — dia menggerakkan.

✍️ Worker

Penulis per-bab. Satu worker menggarap satu bab, paralel dengan worker lain. Tidak perlu jenius — cukup menulis bab sesuai target.

🎩 Manager

Editor akhir. Jalan sekali setelah semua bab selesai — menghaluskan sambungan, menyatukan suara, menambah header bab.

📜 Pakem

Satu sumber kebenaran. Semua aturan gaya, bahasa, struktur VO hidup di sini — Node, Python, dan frontend baca yang sama.

⚡ Paralel

Total waktu = bab paling lambat, bukan jumlah semua bab. 10 bab sekuensial 32 satuan → paralel 7 satuan.

🧮 Cache-aware

System prompt byte-identik tiap bab → prefix caching upstream. Hanya blok dinamis per-bab yang berubah.

Dua engine di UI Narasi: Klasik = jalur lama, bab digarap berurutan satu-satu (stabil). Dalang ⚡ = orchestrator ini — paralel, routing, polish, kontrol penuh. Dokumen ini soal engine Dalang.
End-to-end Flow

Dari klik Approve & Generate sampai narasi jadi

Outline dibuat dulu (jalur biasa). Begitu di-approve, engine Dalang ambil alih:

1

Brief

Satu panggilan LLM bikin narrative brief (tone, arc, motif) dari outline. Jadi konteks lintas-bab.

2

POST /api/narration → 202

Body dikirim ke job orchestrator: topic, chapters[], style, language, mode, worker_model, manager_model, max_workers, polish, orch_mode, rag. Credit di-hold di depan (402 kalau kurang).

node /api/narration → python /narration (Clerk-gated)
3

Shared context (1×)

RAG retrieval + brand lookup sekali untuk seluruh job (bukan per-bab). Hasilnya dipakai ulang semua worker.

orchestrator.context_builder.build_shared_context()
4

Router pilih strategi

Inspeksi bentuk request → scenario A–E. Outline eksplisit → A = static (narrate_chapters).

orchestrator.router · scenario=A settings={…}
5

Fan-out worker (paralel)

1 worker / bab, dibatasi semaphore MAX_WORKERS. Tiap worker: compose() prompt pakem → run_worker(). Checkbox bab nyala via telemetry chN.

asyncio.Semaphore(max_workers) · as_completed
6

Assemble + Manager polish

Bab diurut, dirakit jadi ## Bab N: Judul + isi. Manager (reduce) menghaluskan jadi satu alur — header dipertahankan, fakta & panjang tidak diubah.

_polish_reduce() → synthesize(role)
7

Settle + persist + render

Credit di-settle di biaya asli (blended per-model). Bab tersimpan ke narasi_chapters, buku ke jobs row. UI polling GET /narration/:id → render output.

Polling: UI nge-poll /narration/:id tiap 2 detik → {status, done, total, chapters[{no,state}], output}. Status: running → polishing → done | failed | cancelled. Cancel kapan saja via /narration/:id/cancel.

Worker Fan-out

Bagaimana worker disebar

1 worker = 1 bab (deterministik). Jumlah worker = jumlah bab. Yang variabel cuma durasi tiap worker (tergantung ukuran bab + model). Konkurensi dibatasi MAX_WORKERS (semaphore).

Contoh: 10 bab, MAX_WORKERS = 6 (sliding window)

Bab 12s← gelombang-1 (t=0)
Bab 24s
Bab 33s
Bab 45s
Bab 52s
Bab 64s
Bab 73s← masuk pas Bab 1 kelar
Bab 82s← masuk pas Bab 5 kelar
Bab 94s← pas Bab 3 kelar
Bab 103s← pas Bab 6 kelar
6 worker pertama (langsung jalan t=0) antri → masuk begitu ada slot kosong
  • 6 bab pertama jalan barengan di t=0 (= cap).
  • Bab 7–10 bukan batch kaku — begitu satu worker selesai, bab berikut langsung mengisi slot (sliding window).
  • Peak concurrency = 6, tidak pernah lebih.
  • Total 7 satuan vs 32 kalau berurutan → ~4,5× lebih cepat.
Paralel hemat WAKTU, bukan BIAYA. 10 bab tetap = 10 panggilan worker. Naikin max_workers mempercepat (lebih banyak bab barengan) tapi menambah risiko rate-limit upstream.
Single Source of Truth

Pakem & prefix caching

Semua aturan gaya/bahasa/VO hidup di paket pakem. Tiap bab, compose() menghasilkan dua bagian:

🧊 System prefix — STABIL

Kanon gaya + aturan + outline lengkap. Byte-identik di semua bab dalam satu job → di-cache upstream (prefix caching). ~6.000 token yang tidak dibayar ulang per bab.

🎯 User block — DINAMIS

Cuma fokus bab ini: ## THIS CHAPTER: judul + summary + word target + “story so far” (prev_tail) untuk kontinuitas. Kecil (~50–500 token).

# Terbukti live — hash system prefix identik 3 bab:
system hash ch0: 8fce9bf2036180f2
system hash ch1: 8fce9bf2036180f2
system hash ch2: 8fce9bf2036180f2
SYSTEM PREFIX IDENTICAL: True   # → cacheable
user blocks DISTINCT  : True   # fokus + prev_tail beda tiap bab
Katalog publik dari pakem: 14 gaya narasi + 20 bahasa (termasuk Basa Jawa, Sunda, Bali) di /api/narration/styles & /languages — frontend baca ini, bukan list hardcoded.

Panduan Model

Model efisien: worker vs manager

Prinsipnya beda peran, beda kebutuhan. Worker butuh cepat & ringkas (jalan paralel, 1/bab). Manager butuh pintar (jalan 1×, nentuin kohesi).

✍️ Worker — utamakan KECEPATAN + ringkas

Worker cukup “nulis bab sesuai target”. Tidak perlu model termahal. Yang penting: cepat, tidak boros token, taat panjang. Hindari reasoning-model sebagai worker — token “mikir”-nya bikin lambat & mahal walau hasil akhirnya pendek.

🎩 Manager — utamakan KUALITAS

Jalan sekali di atas seluruh buku — di sinilah model kuat berbayar. Tapi mahal: di run nyata, manager Claude-Opus = 85% dari total biaya. Pilih sesuai ambisi kualitas vs budget.

Metering

Biaya & credit

Credit dipotong dari biaya asli blended — menjumlahkan biaya tiap call lintas model (worker + manager bisa beda harga), bukan menaksir semua token di satu harga.

# Konversi (credit_catalog):
1 credit  ≡  $0.01 upstream cost          # CREDIT_USD_VALUE
credits   =  ceil( usd × MARGIN / 0.01 )       # MARGIN=1.0, min 1

# Contoh run 3 (gemini-pro worker + claude-opus manager):
3 worker gemini-2.5-pro  = $0.0915
1 manager claude-opus    = $0.1113
                  blended = $0.2028  →  usd_to_credits = 21 credit ✓
Worker = penentu KECEPATAN, Manager = penentu BIAYA. Di run nyata, manager Claude-Opus = 85% total biaya. Mau murah? Pakai manager murah (gpt-4o-mini) atau polish=none. Paralel hemat waktu, bukan token.
Invariant

Aturan & jaminan engine

🛡️ Never-raise

run_worker & router tidak pernah melempar exception ke caller. Kegagalan jadi marker {ok:false}, bukan crash.

🩹 Graceful degradation

Bab yang gagal → placeholder berlabel, buku tetap terakit. Polish di-skip kalau ada bab gagal (biar editor nggak bingung).

♻️ RAG sekali per job

Retrieval + brand lookup sekali di build_shared_context, dipakai ulang semua bab — bukan per-bab.

🧊 Prefix stabil

System prompt byte-identik tiap bab dalam satu job → cache-friendly. Cuma blok dinamis yang berubah.

📑 Header dipertahankan

Polish wajib menjaga tiap ## Bab N + tidak memotong panjang + tidak mengubah fakta.

💳 Settle akurat

Credit di-settle di biaya asli per-model (blended). Hold di depan (402 kalau kurang), refund kalau gagal/cancel.
