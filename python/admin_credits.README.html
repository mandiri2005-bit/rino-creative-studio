admin_credits.py

Beri & cek kredit user secara manual — untuk masa bootstrap saat Stripe belum dipakai. Lewat lapisan durable yang sama dengan metering live (Postgres ledger + cache Redis), jadi saldonya konsisten dengan yang dipotong tiap operasi.

01 Dua perintah

grant
Tambah (atau kurangi) kredit untuk satu tenant. Bisa idempoten lewat --op-id.
balance
Lihat saldo live + durable + hasil rekonsiliasi untuk satu tenant.
cara menjalankan — dari host (Docker)
# beri 5.000 kredit ke user berdasarkan email
docker exec rino-creative-studio-python-api-1 \
  python /app/admin_credits.py grant --email user@contoh.com --credits 5000

# cek saldo
docker exec rino-creative-studio-python-api-1 \
  python /app/admin_credits.py balance --email user@contoh.com

# di Railway (produksi)
railway run python /app/admin_credits.py grant --tenant <uuid> --credits 5000

02 Semua parameter

Wajib pilih salah satu dari --email atau --tenant untuk menunjuk user. Sisanya hanya untuk grant.

FlagPerintahWajibArti
--emailgrant · balance salah satu* Email user. Di-resolve ke tenant lewat tenant_id_by_email(). *wajib salah satu dengan --tenant.
--tenantgrant · balance salah satu* tenant_id (UUID) langsung — dipakai kalau kamu sudah tahu id-nya (mis. dari Sentry/log). Lebih pasti daripada email.
--creditsgrant ya Jumlah kredit ditambah. Boleh negatif untuk menarik kembali (mis. --credits -500). 1 kredit = $0,01.
--op-idgrant opsional Kunci idempotensi. Grant dengan op-id yang sama untuk tenant itu hanya masuk sekali. Lihat bagian 03. Kosongkan = tiap jalan menambah lagi.
--reasongrant opsional Label di ledger: admin_adjust (default) · topup · monthly_grant. Nilai lain otomatis jadi admin_adjust.

03 --op-id  kenapa penting

Idempotensi = aman dari kredit dobel

Ledger kredit punya batasan unik (tenant, op_id). Artinya satu op_id hanya bisa "mendarat" sekali per user. Berguna untuk: jalan ulang tak sengaja, retry skrip, atau kode promo yang setiap user cuma boleh klaim sekali.

✕ tanpa --op-id
grant --email a@x.com --credits 500
grant --email a@x.com --credits 500
→ saldo +1000 (dua-duanya masuk)
✓ dengan --op-id
grant ... --credits 500 --op-id promo-LAUNCH
grant ... --credits 500 --op-id promo-LAUNCH
→ saldo +500 (yang kedua diabaikan)

Tips kode promo: pakai op-id yang menyertakan nama kampanye, mis. --op-id LAUNCH50. Karena uniknya per tenant+op-id, tiap user otomatis cuma bisa klaim LAUNCH50 satu kali.

04 Resep cepat

copy-paste
# kredit beta untuk user baru
... admin_credits.py grant --email beta@user.com --credits 5000 --reason topup

# kode promo (idempoten — aman dijalankan berkali-kali)
... admin_credits.py grant --email a@x.com --credits 2000 --op-id LAUNCH50

# tarik kembali kredit (koreksi) — pakai angka negatif
... admin_credits.py grant --tenant <uuid> --credits -1000

# cek saldo + rekonsiliasi (live vs durable vs ledger)
... admin_credits.py balance --tenant <uuid>

05 Yang perlu diingat

Gerbang aktif

Metering ON di bootstrap untuk melindungi dari cost-bleed. Matikan total dengan METERING_ENABLED=false lalu docker compose up -d.

Nilai kredit

1 kredit = $0,01 biaya upstream (CREDIT_USD_VALUE). Jadi 5.000 kredit ≈ $50 pemakaian.

Alokasi free

User baru otomatis dapat 100 kredit (≈$1). Kecil untuk beta — naikkan TIER_MONTHLY_CREDITS["free"] di credit_catalog.py bila perlu.

Sumber kebenaran

Saldo durable di Postgres credit_balances/credit_ledger; Redis cuma cache cepat. balance menampilkan keduanya + cek drift.
